Agentic Projects – Generic Setup &
Execution Guide (2-Week Track)
This document applies to all teams (single-agent + tools and multi-agent). It standardizes
environment setup, project workflow, required engineering practices, and a recommended
“agentic runtime” stack.
0. What You’re Building (Definition)
An agentic application is not a chatbot. It is a system with:
● Explicit goal(s)
● A state machine or workflow graph
● Tool calls / actions against an environment
● Logged decisions and measurable outcomes
● UI visibility into state + actions + messages
1. Team Workflow (How to Proceed)
Week 1 (Build the backbone)
1. Freeze scope (Day 1): inputs, outputs, constraints, metrics, test scenarios.
2. Design doc (Day 1–2): architecture, state machine, tool schemas, UI plan.
3. Implement environment + state (Day 2–3): deterministic simulator/state store.
4. Implement agents + tools (Day 3–5): structured outputs, tool wrappers.
Week 2 (Make it reviewable)
5. UI + observability (Day 6–9): state machine UI, transcripts, metrics dashboard.
6. Evaluation harness (Day 8–10): 10 scenarios, seed reproducibility, baseline.
7. Hardening (Day 11–12): failure cases, rate limits, timeouts, guardrails.
8. Demo readiness (Day 13–14): scripted demo run + replay + documentation.
2. Required Technical Components (All Projects)
2.1 State machine
● Must be explicit in code (enum/state table) and visible in UI.
● Must log: previous_state → event → next_state.
2.2 Observability
● Every run must have a Run ID.
● Log: agent name, prompt/inputs, tool calls, tool I/O, state transitions, timestamps.
● Store logs as JSON Lines (recommended) or SQLite.
2.3 Reproducibility
● Every run must accept a seed.
● Simulation/data generator must be seedable.
● Teams must demonstrate: same seed → same outputs.
2.4 Metrics
● Minimum 3 quantitative metrics.
● Minimum 10 scenarios (synthetic acceptable).
● Must show baseline comparison (single-agent baseline for multi-agent; no-tool baseline
for single-agent track).
2.5 UI (mandatory)
UI must show:
● Current state + state transition history
● Which agent is active
● Agent messages/transcript
● Tool calls (tool name + JSON input/output)
● Metrics dashboard
● Run controls (Start/Reset/Seed)
3. Recommended Local Tech Stack (Zero Paid Tools)
Choose one stack and stay consistent.
Option A (Fastest): Python + Streamlit
● Backend + orchestration: Python
● UI: Streamlit
● Storage: SQLite or JSONL
Option B (More “product”): FastAPI + React
● Backend: FastAPI
● UI: React
● Storage: SQLite
No paid services. Use local execution only.

● Node.js 18+ (only if React UI)
Recommended

5. “Agentic Environment” Choices (MCP / A2A / Simple
Local Tools)
You mentioned MCP and n8n. MCP is a protocol for structured tool access. n8n is a workflow
automation platform.
5.2 n8n – Optional (Workflow-Oriented Agent Orchestration)
Use n8n when:
● You want visual workflow orchestration
● You prefer low-code tool chaining
● You want to expose HTTP endpoints as tools
What n8n is:
● An open-source workflow automation tool that allows node-based orchestration.
When to use it in these projects:
● Single-agent + tools track (for orchestrating tool chains)
● Multi-agent prototypes where each agent is triggered as a node
When NOT to use it:
● If it hides state transitions from your UI
● If it becomes the only “logic layer” without visible state machine implementation
Important: Even if using n8n, you must still implement:
● Explicit state machine logic
● Structured logging
● Tool I/O visibility in your UI
For most intern projects, a local Python orchestration layer is sufficient. n8n is optional, not
required.
5.3 Simple Local Tool Layer (No MCP)
If MCP feels heavy:
● Define tools as functions with JSON schema (Pydantic)
● Enforce timeouts
● Log inputs/outputs
● Keep tools deterministic
This is acceptable as long as tool I/O is visible in UI.
7. Configuration & Secrets
● Put the LLM API key in environment variables (never commit).
● Provide .env.example with required keys.
● Provide a config.json per run (seed, parameters, scenario).
8. Guardrails (Required)
● Tool allowlist (agent can call only approved tools)
● Max steps per run (prevents infinite loops)
● Timeouts on tool calls
● Output validation (schema validation + constraints)
● Human override mode (Stop / Reset)
9. Demo Expectations
Every team should be able to demo in 5–7 minutes:
1. Show architecture diagram + state machine
2. Start a run with a visible seed
3. Show agent/tool actions in UI
4. Show state transitions
5. Finish run and show metrics
6. Re-run same seed to show reproducibility
7. Run one failure scenario (unexpected event) and show graceful handling
